## 1. 产品定位

**一句话定义**：面向特定中文学者的学术动态监控仪表盘。

**核心价值**：张老师需要持续追踪若干关键学者的最新学术产出（论文、政策研究、公共发言），以便在写作和研究决策时快速掌握领域动态。不是文献管理工具，不是搜索引擎——是一个"谁最近说了什么"的**监控面板**。

**使用场景**：
- 每周打开一次，浏览追踪学者的最新产出
- 写作前检查：某位学者最近有没有相关新观点
- 发现跨学者的观点交锋或议题共振

> **扩展机制**：学者清单应放在独立配置文件（`config/scholars.yaml`），后续可通过编辑该文件增减。新增学者时，需要指定其数据源偏好——国内学者默认走知网+公众号，国际学者走 arXiv + Google Scholar。

### 2.2 数据源决策规则（关键）

这是前期踩过的最大坑，接手的人必须记住：

```
if 学者.主要语言 == "中文":
    数据源 = 知网(CnkiSpider) + 微信公众号(搜狗微信搜索) + 官方政策网站
    禁止使用 = arXiv, OpenAlex, Semantic Scholar, Google Scholar
else:
    数据源 = arXiv + OpenAlex + Semantic Scholar + Google Scholar
```

**原因**：国内社科学者的学术产出不在英文数据库中。用 arXiv 搜江小涓等于什么也搜不到。

## 3. 产出类型与分类

### 3.1 产出分类体系

| 类别 | 说明 | 示例 |
|------|------|------|
| **学术论文** | 期刊/会议论文，来源：知网 | 《经济研究》2025年第3期 |
| **政策报告** | 政府委托研究报告、政策建议 | 国务院发展研究中心报告 |
| **公众号文章** | 学者本人或所在机构公众号发文 | "江小涓"公众号《数字时代的服务经济》 |
| **媒体访谈** | 媒体专访、公开演讲整理稿 | 财新专访、CC讲坛演讲 |
| **政策文件/批示** | 学者参与起草或署名的官方政策文件 | 中央深改委文件 |

### 3.2 分类逻辑

- 知网抓取 → 自动标为「学术论文」
- 公众号抓取 → 自动标为「公众号文章」
- 政策网站抓取 → 自动标为「政策文件」
- 其他来源 → 需人工标注类别

## 4. 功能需求

### 4.1 核心功能（MVP 必须实现）

1. **多源采集**
   - 从知网、微信公众号、政策网站采集指定学者的最新产出
   - 去重（同一篇文章可能出现在多个来源）
   - 按时间倒序排列

2. **仪表盘展示**
   - 一张页面看完所有追踪学者的最新产出
   - 支持按学者筛选、按产出类型筛选
   - 每条产出显示：标题、学者、来源、日期、摘要、原文链接

3. **自动更新**
   - 可手动运行采集脚本
   - 理想情况：GitHub Actions 定时（如每周一早上）自动运行采集 → 自动更新页面

### 4.2 进阶功能（v2）

- **观点提取**：对长文章自动提取核心论点标签
- **交叉关联**：发现多位学者同时讨论同一议题
- **邮件/飞书推送**：每周摘要推送到张老师

### 4.3 不做的事情

- ❌ 不做全文翻译
- ❌ 不做深度内容分析（那是张老师自己的事）
- ❌ 不做文献管理（有 Zotero）
- ❌ 不做引用关系图谱

## 5. 数据结构

每条产出记录必须包含以下字段：

```yaml
id: "唯一ID"
scholar: "学者姓名"
title: "产出标题"
source_type: "学术论文 | 公众号文章 | 媒体访谈 | 政策文件 | 政策报告"
source_name: "具体来源名称，如《经济研究》2025(3)"
date: "发表日期 YYYY-MM-DD"
url: "原文链接（必须有，不能为空）"
abstract: "摘要或首段（≤200字）"
tags: ["关键词1", "关键词2"]
fetched_at: "采集时间戳"
```

**关键约束**：
- `url` 字段**不能为空**——张老师需要原始出处做论文引用和查证
- `abstract` 可以是原文首段截取，不需要 AI 生成
- `tags` 初期留空或手动标注，不做自动标注

## 6. 技术方案建议

### 6.1 架构

```
采集层 (Python)
  ├── collect_cnki.py     → 知网采集（爬虫/CnkiSpider API）
  ├── collect_wechat.py   → 微信公众号采集（搜狗微信搜索）
  ├── collect_policy.py   → 政策网站采集
  └── collect_media.py    → 媒体访谈采集（可选）

存储层
  └── data/archive.json   → 所有产出汇总（单文件 JSON，便于部署）

展示层
  └── render.py           → 读取 JSON，生成自包含 HTML
  └── index.html          → 部署到 GitHub Pages
```

### 6.2 部署

- **推荐方案**：GitHub Pages（静态网站）+ GitHub Actions（定时采集）
- **流程**：Actions 定期运行采集脚本 → 更新 archive.json → 重新生成 HTML → 自动部署
- **备选方案**：本地脚本手动运行，HTML 手动上传到任意静态服务器

### 6.3 技术选型

- 采集脚本：Python（requests + BeautifulSoup / 各平台 SDK）
- 数据存储：JSON（单文件，≤10MB，全量数据）
- 前端：纯 HTML/CSS/JS（自包含单文件，不依赖框架和 npm）
- 部署：GitHub Pages（免费、自动 HTTPS）

## 7. UI/UX 规格

### 7.1 设计原则

- **一屏看完**：所有追踪学者的最新产出，一页全展示
- **信息密度高**：不要大留白，不要卡片式，紧凑排列
- **暗色主题**：适合长时间阅读（参考 GitHub Dark）
- **移动端可读**：响应式布局，手机也能看

## 8. 文风与数据质量要求

### 8.1 数据质量标准

- **真实**：每一条产出都必须有可验证的原文链接
- **完整**：标题、作者、日期、来源、链接五个字段缺一不可
- **去重**：同一篇文章只出现一次
- **时效**：采集时间与实际发表时间的延迟不超过 48 小时

### 8.2 绝对禁止

- ❌ 虚构产出条目（mock 数据只能用于开发测试，绝对不能进入生产环境）
- ❌ 链接为空或链接指向不存在页面
- ❌ 用 AI 生成摘要替代原文摘要
- ❌ 把无关内容（如通用行业新闻）混入学者的产出列表

|------|------|-----------|

| **M3：数据存储** | 实现 JSON 存储 + 去重逻辑 | 0.5 天 |

| **M5：部署** | GitHub Pages + Actions 自动更新 | 0.5 天 |
